# Архитектура

## Обзор

```
ИСТОЧНИК КАТАЛОГА                 СБОРКА (build-time)                 РЕЗУЛЬТАТ (статика)
(у клиента)                       (твоя машина / CI)                  (домен клиента)

YML / CSV / Google Sheets  ──►  Adapter → канон → нормализация  ──►  HTML/CSS/JS (Astro SSG)
                                        │                            + data/catalog.json
                                        ├─ CLIP: эмбеддинги картинок  + data/embeddings.json
                                        └─ GigaChat: тексты + SEO     + public/models/ (CLIP)

RUNTIME (в браузере у покупателя):
  поиск по фото / семантический поиск  →  CLIP в браузере + косинус по embeddings.json
  заказ                                →  POST на общий релай → email + Telegram клиента
```

Ключевая идея: **вся «тяжёлая» работа — на сборке**. В готовом сайте нет серверной логики, кроме крошечного общего релая заказов. Это и даёт нулевую стоимость эксплуатации и отсутствие подписки.

## 1. Слой данных

Источник правды по товарам — **не CRM**, а учётная система / CMS клиента (1С, МойСклад, Битрикс, InSales…). Почти все они умеют отдавать **YML** (формат Яндекс.Маркета). Поэтому мы не пишем интеграцию под каждую систему, а делаем:

- **Канонический формат товара** — единый внутренний `Product` (см. `DATA-MODEL.md`). Весь сайт видит только его.
- **Адаптеры** — маленькие модули «привести источник к канону». В MVP: **YML/CSV** и **Google Sheets**. Этого хватает на ~90% среднего бизнеса.
- **Capability-флаги** — адаптер декларирует, что умеет (есть остаток? фото? варианты?), а UI деградирует, если поля нет.

Синк — **pull по расписанию** (`npm run sync`, либо крон/вебхук в CI). Каталог мебели меняется медленно, реалтайм не нужен.

## 2. Пайплайн сборки

1. `sync` — адаптер тянет источник → `toCanonical()` → `data/catalog.json`.
2. `embed` — CLIP считает эмбеддинг главного фото каждого товара → `data/embeddings.json` (см. `PHOTO-SEARCH.md`). Векторы L2-нормированы.
3. `enrich` (опц.) — GigaChat полирует описания и генерит SEO для товаров с пустым/сырым текстом; результат кешируется по хэшу (не пересчитываем неизменное).
4. `astro build` — генерит статические страницы из `catalog.json`; `embeddings.json` и веса CLIP кладутся в `public/`.

## 3. Поиск по фото и тексту (фишка)

Один движок эмбеддингов закрывает сразу несколько сценариев: поиск по фото, семантический поиск текстом, блок «похожие товары», «соберите образ». Считается **в браузере** (CLIP через transformers.js/ONNX), данные пользователя никуда не уходят — это удобно и для 152-ФЗ. Подробно — `PHOTO-SEARCH.md`.

## 4. Тексты и SEO (GigaChat)

GigaChat вызывается **только на сборке**, из твоего пайплайна, **одним твоим платным ключом (ИП/юрлицо)** на всех клиентов. В готовом статическом сайте GigaChat не вызывается — там лежит готовый текст. Клиенту аккаунт не нужен. Подробно — `GIGACHAT.md`.

## 5. Заказы

Сайт статический, поэтому отправку email/Telegram делает **один общий serverless-релай** (Yandex Cloud Function, scale-to-zero) на всех клиентов:

```
форма заказа (браузер) → POST {tenant, order} → релай → email + Telegram нужного клиента
```

- Роутинг по `tenant` из `site.config.ts` (email + Telegram chat id).
- Стоимость near-zero: платишь доли копейки за вызов, объём заказов низкий. Не вводит ни per-client backend, ни подписку.
- **Абсолютный fallback** без сервера: deep-link (`wa.me` / Telegram / `mailto:`) с предзаполненным заказом — но релай надёжнее (заказ не теряется).

## 6. Хостинг и мульти-тенантность

- **Статика сайта** — на домене/хостинге клиента (или дешёвый Object Storage). Yandex Cloud клиенту подключать НЕ нужно.
- **Общая инфра у тебя** (одна на всех): релай заказов. (CLIP — в браузере, отдельного сервиса не требует в базовом тире.)
- **Изоляция данных** между клиентами — логическая: отдельный конфиг, свой `public/models/`, свой каталог. Без БД и без отдельных аккаунтов на клиента.
- **152-ФЗ:** данные покупателя обрабатываются в браузере (поиск) или проходят через RU-релай; веса CLIP хостятся на домене клиента (не на зарубежном CDN).

## 7. Фиче-тоглы и graceful degradation

Один код обслуживает всех. Поведение определяется конфигом и наличием данных в фиде:

| Условие | Поведение |
|---------|-----------|
| Нет картинок в фиде | Нет поиска по фото и блока «похожие» |
| Нет остатка | Статус «уточняйте наличие» вместо «в наличии» |
| Нет вариантов | Карточка без переключателя ткани/цвета |
| Нет 3D/AR-моделей | Кнопка AR скрыта |
| Тонкий фид | Базовые фильтры из тех атрибутов, что есть |

## 8. Что отложено (спроектировано, не строим в MVP)

- **AR/3D в карточке** — веб-компонент `<model-viewer>` (iOS Quick Look / Android Scene Viewer). Включается, если у клиента есть GLB/USDZ. Опциональный тир.
- **Прямой API** (1С / МойСклад / Битрикс) для живого остатка — premium edge-case, тянет за собой runtime + БД.
- **Админка** вместо Google-таблицы — отдельный платный тир.
- **Конфигуратор / планировщик комнаты** — кастомный проект под клиента, не шаблонная фича.
- **Serverless CLIP** (scale-to-zero) — для огромных каталогов или мгновенного поиска без загрузки модели. Платный тир.

Эти пункты учтены в интерфейсах (canonical-формат и capability-флаги их переживают), но в MVP не реализуются.
