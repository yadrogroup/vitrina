# Поиск по фото и семантический поиск

Сигнатурная фича. Работает **в браузере, без сервера** — это и бесплатно в эксплуатации, и удобно по 152-ФЗ (фото пользователя никуда не отправляется).

## Идея: один движок — несколько сценариев

CLIP кладёт картинки и текст в **одно векторное пространство**. Поэтому одна модель закрывает:

- **поиск по фото** — эмбеддинг загруженного фото → ближайшие товары;
- **семантический поиск текстом** — эмбеддинг текста запроса → товары (работает даже на мусорных описаниях фида, т.к. матчим с картинками);
- **«похожие товары»** в карточке — эмбеддинг товара → соседи;
- **«соберите образ» / кросс-селл** — по визуальной близости.

## Как устроено

### На сборке (build-time)
1. Для каждого товара берём главное фото → прогоняем через **image-энкодер CLIP** → вектор.
2. **L2-нормируем** вектор (тогда косинусная близость = скалярное произведение, дёшево).
3. Пишем `data/embeddings.json`:
   ```json
   { "model": "Xenova/clip-vit-base-patch32", "dim": 512,
     "items": { "<productId>": [0.01, -0.03, ...] } }
   ```
4. Веса CLIP кладём в `public/models/` (на домене клиента, НЕ зарубежный CDN).

### В рантайме (браузер)
- **Текстовый запрос:** грузим только text-энкодер CLIP → эмбеддинг строки → скалярное произведение с `items` → топ-K → карточки. Лёгкий, можно держать включённым в строке поиска.
- **Поиск по фото:** по клику ленивая загрузка image-энкодера → препроцесс картинки (resize/normalize) → эмбеддинг → топ-K. Фото остаётся в браузере.
- Сравнение (cosine/dot) — обычный JS-цикл по массиву. Для сотен–тысяч товаров это <100 мс, никакой векторной БД не нужно.

```ts
// псевдокод ранжирования
const q = normalize(await clip.embed(input));        // фото или текст
const ranked = Object.entries(embeddings.items)
  .map(([id, v]) => [id, dot(q, v)])                 // v уже нормирован на сборке
  .sort((a, b) => b[1] - a[1])
  .slice(0, K);
```

## Библиотеки и модель

- **transformers.js (`@huggingface/transformers`)** или **ONNX Runtime Web** — инференс CLIP в браузере (WASM, при наличии — WebGPU).
- Модель: **`Xenova/clip-vit-base-patch32`** (квантованная, int8). Image+text энкодеры можно грузить раздельно.
- Веса самохостим: `npm run fetch-model` кладёт их в `public/models/`, в коде указываем локальный путь (`env.allowRemoteModels = false`).

## Производительность и UX

- Модель — десятки–~сотня МБ; грузится **лениво при первом использовании** и кешируется (Cache API / браузерный кеш). Дальше — мгновенно.
- Состояния dropzone: `idle → загрузка модели… → анализирую фото… → результаты`. Текстовый поиск можно подключить раньше (он легче).
- `embeddings.json` для типичного каталога мал; при больших каталогах можно квантовать векторы (int8) и грузить чанками.
- Деградация: нет картинок у товара → он не участвует в визуальном поиске; нет WebGPU → fallback на WASM (медленнее, но работает).

## 152-ФЗ

- Фото и запрос обрабатываются **в браузере**, на сервер не уходят → персональных данных не покидает устройство.
- Веса модели — на домене клиента (RU), не на зарубежном CDN.

## Опциональный тир (не в MVP): serverless CLIP

Для очень больших каталогов или чтобы не грузить модель в браузер — вынести инференс в **Yandex Cloud Function/Container со scale-to-zero**: браузер шлёт фото → функция возвращает вектор → сравнение всё равно можно делать в браузере по `embeddings.json`. Платится только за фактические вызовы. Это апсейл, а не дефолт.
